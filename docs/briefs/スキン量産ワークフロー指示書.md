# スキン量産ワークフロー指示書（Claude Code / Sonnet運用）

キャッテリーごとに「別のサイト」に見えるデザインを、ブリーフを書くだけで量産するための手順書。
道具は **VS Code + Claude Code拡張（モデル: Sonnet）**。規約は **CLAUDE.md** が毎セッション自動で効く。

## 0. 全体像

```
パターン ＝ スキン（色×書体×角丸×質感） × ヒーローバリアント（レイアウト） × hstyle（見出し装飾）

あなた：ブリーフを書く（5〜10分）
   ↓
Claude Code：skin-{name}.scss 作成 → functions.php登録 → ビルド（10〜20分）
   ↓
あなた：ローカルWPのカスタマイザーで切替確認 → 微調整指示 → レビュー（20〜30分）
   ↓
スクショをカタログへ ＋ git commit
```

1スキンあたり実働60〜90分。**3スキン系統×4バリアント＝12パターン**が最初の目標。

## 1. 初回セットアップ（1回だけ・30分）

1. **VS CodeにClaude Code拡張を導入**：拡張機能マーケットプレイスで「Claude Code」を検索→インストール→Anthropicアカウントでサインイン。モデル設定でSonnetを選択
2. **ワークスペース**：LocalのWordPress内のテーマフォルダ（`…/app/public/wp-content/themes/catelabo-base`）をVS Codeで開く。ここを作業リポジトリにすると、**保存→ブラウザ更新で即プレビュー**できる
3. **ファイル配置**：テーマフォルダ直下に次を置く
   - `CLAUDE.md`（納品済みのファイル）
   - `docs/テーマ開発指針.md`（納品済みの指針をコピー）
   - `docs/briefs/_テンプレ.md`（納品済みの「**スキンブリーフ_テンプレ.md**」を、この場所にこの名前でコピーして配置）
4. **動作確認**：VS Codeのターミナルで `sass assets/scss:assets/css --no-source-map` が通ること
5. **git**：`docs/` `CLAUDE.md` もコミット対象に含める（規約の変更履歴も資産）。運用はシンプルに「1スキン＝1コミット」。壊れたら `git checkout` で戻せる安心が量産の土台

> 補足：顧客サーバーへアップするzipに `CLAUDE.md` や `docs/` が含まれても実害はないが、気になる場合はzip作成時に除外する。

## 2. 新スキン制作フロー（メインの作業）

### Step 1｜ブリーフを書く
`docs/briefs/_テンプレ.md` をコピーして `docs/briefs/skin-{name}.md` を作成し、**テンプレの構造を残したまま**記入（空欄可＝Sonnetが提案してくる）。
**色は言葉でもHEXでも可**。参考画像は `docs/briefs/refs/skin-{name}/` に配置（InstagramはスクショでN枚。一般サイトはURLでも可）。
**望むレイアウトが頭にあるなら、ラフでいいのでモック画像を描いて置くのが最速**（skin-yukiの仕上げで実証済み。言葉での説明より数倍速い）。

### Step 2｜Claude Codeに依頼（開始プロンプトv4をコピペ）

```
CLAUDE.md と docs/テーマ開発指針.md（特に§9の落とし穴）、.claude/skills/ の
design-taste-frontend / high-end-visual-design / redesign-existing-projects を
読んでから作業してください。
品質見本: assets/scss/skins/skin-yuki.scss。
「レイアウトの文法ごと変える」この水準が合格ラインです。

新しいスキンを1つ追加します。ブリーフ: docs/briefs/skin-{name}.md
（refs/ フォルダがあれば参考画像・モックも見ること）

進め方（この順番で・省略禁止）:
0. think hard で取り組むこと
1. 【研究】ブリーフの雰囲気に合う「今どきのレイアウトの文法」を列挙する
   （縦書き／巨大タイポ／ゴースト文字／番号付きセクション／脱カード／
     オーバーラップ／額装／グリッド崩し など）
2. 【提案】方向性を2案。それぞれ「ヒーロー構成・見出し構成・カードの見せ方・
   動きの署名」を3行ずつで示し、推し案と理由を述べて私の承認を待つ
3. 【実装】承認後に skins/skin-{name}.scss を作成
   ・CSSによる構造再構成OK（grid組替え・writing-mode・疑似要素タイポ等）
   ・HTML/PHP/JSには触れない。スマホ表示と reduced-motion を壊さない
   ・上書きするセレクタはベース側の全プロパティを確認し、必要な打ち消しを明示する
   ・提出前に自問する：「これは skin-default の色替えに見えないか？」
     見えるなら作り直してから提出
4. functions.php 登録 → ビルド → 完了報告
   （CLAUDE.mdの報告要件 ＋ SP時にどういう構成になるかの説明を含める）
```

### Step 3｜差分を検品する
VS Code上でClaudeの変更差分を必ず読む。見るポイント：
- 触られたファイルが `skins/skin-{name}.scss` と `functions.php` （＋ビルド出力）だけか
- スキンファイルの中身が `:root { --token: … }` 中心か

### Step 4｜スクリーンショット往復で微調整する（ここが仕上げの本体・5〜10往復を想定）
カスタマイザーでスキン・バリアント・hstyle・モーションを組み合わせて確認（**必ずスーパーリロード**。ビルドしてもブラウザは古いCSSをキャッシュする）。PC・スマホ幅の両方。
一発で仕上がらない前提で、次の型のフィードバックを回す：
- **スクショを貼り「どの要素を・何px（rem）・どちらへ」**で指示する（例:「p-hero__eyebrowのmargin-topを4remに」「写真をウィンドウ端から20px内側に」）
- 望む姿が明確なら**モック画像を貼る**（最強。skin-yukiの決着はこれだった）
- 挙動が意図と違うときは「**制御する場所が間違っていないか、ベース側の指定を確認して**」と促す（min-height残置などの定番事故を自分で掘らせる）
- 色・書体・角丸の調整は**トークンの言葉で**指示すると速い

### Step 5｜レビューチェックリスト（実施者＝あなた。Sonnetの完了報告を検収する）

**A. Sonnetの報告を照合する項目**（報告値を確認。怪しければ自分で再実行）
- [ ] `sass` ビルド成功
- [ ] 生HEXチェック0件。テーマフォルダのターミナルに下記を貼って実行し、**何も表示されなければ合格**（表示された行＝違反箇所として差し戻す）
      `grep -rn "#[0-9a-fA-F]\{3\}" assets/scss --include="*.scss" | grep -v _tokens | grep -v skins/`
      ※「トークンとスキン以外のSCSSに生の色コード(#fff等)が紛れていないか」を探す＝規約「色は必ずトークン経由」の機械検査
- [ ] 本文・見出しのコントラスト比4.5:1以上（Sonnetの報告値を確認）
- [ ] HTML・テンプレートPHP・JSに変更が入っていない（`git diff --stat` で変更ファイルがSCSSとfunctions.phpだけであること）

**B. あなたの目と手で確認する項目**
- [ ] スマホ実機（またはデベロッパーツール375px）で崩れなし
- [ ] 既存スキン（default）に切り替えて**元の見た目が壊れていない**こと
- [ ] 縦書き見出しの折返し位置が手動改行どおり／要素の重なりなし
- [ ] OSの「視差効果を減らす」ONで出現モーションが無効化される（reduced-motion）
- [ ] 指針§9の落とし穴に該当する箇所がないか流し読みで最終確認

### Step 6｜カタログ化とコミット
- TOPのスクショを保存：`catalog/{skin}-{variant}-{hstyle}.png`（例: yuki-minimal-bottom.png）
- `git add -A && git commit -m "skin: yuki を追加"`

このスクショ集がそのまま**「12のデザインからお選びいただけます」という営業資産**になる。

## 3. 新ヒーローバリアント制作フロー

スキン（CSSのみ）では作れない「HTMLごと必要な新形」（複数写真のヒーロー等）を増やすとき。
**スキン制作とは別セッションで、以下のプロンプトを単体で使う**（Step 2のプロンプトに追記するのではない）。
スキン制作中にSonnetが「これはバリアント昇格が必要」と報告してきたら、そのスキンをいったん完成させてから、新しいセッションでこちらを実行する。

```
CLAUDE.md と docs/テーマ開発指針.md（特に§5クラス契約と§9の落とし穴）を読んでから
作業してください。think hard で取り組むこと。

ヒーローの新バリアント p-hero--{name} を追加します。
イメージ: {言葉で説明。参考画像・モックは docs/briefs/refs/ に置いたので見ること}

手順:
1. 実装方針（HTMLに何を足すか・契約クラスへの影響・SP時の構成）を先に説明し、
   私の承認を待つ。いきなりコードを書かない
2. 承認後: assets/scss/object/project/_p-hero.scss にバリアント1ブロック追加、
   inc/theme-options.php の catelabo_hero_variants() に登録
3. front-page.php の変更が必要な場合は、既存の契約クラス
   （__body __eyebrow __title __divider __text __cta __media）を保ったまま最小限に
4. ビルドして完了報告
   （既存バリアント split / photo / minimal が壊れていないことの確認を含む）
```

バリアントはHTML構造に触れる分、スキンより慎重に。**手順1の「先に説明→承認」を必ず守らせる**こと。

## 4. 12パターンの組み立て例

| スキン系統 | 色×書体の方向 | ×バリアント |
|---|---|---|
| 陽だまり（default） | クリーム×モス×明朝 | split / photo / minimal / 新規 |
| 雪 | 白×ブルーグレー×細明朝 | split / photo / minimal / 新規 |
| 夜会 | 濃紺×ゴールド×明朝 | split / photo / minimal / 新規 |

3スキン×4バリアント＝12。hstyleと角丸・質感（線/影）の振り分けで、同系統内の差も作れる。
顧客案件では「近い系統のスキンをコピーして色だけ差し替え」から始めると1件30分まで縮む。

## 5. Claude Codeを使う上での約束（初心者向け）

1. **1セッション1テーマ**。「ついでにあれも」を足すと検品不能になる
2. **差分は必ず読む**。読んで分からない変更は「この変更はなぜ必要？」と聞く。答えが規約で説明できなければ差し戻す
3. ファイル編集や コマンド実行の**許可ダイアログ**が出たら、対象を見て許可（`rm` や指示外のフォルダは拒否）
4. うまくいかないときは、直させるより **`git checkout -- .` で戻してブリーフを直す**方が速いことが多い
5. 規約違反を見つけたら、その場で直させるだけでなく「CLAUDE.mdのどの条項に反するか」を言わせる（再発防止になる）

## 6. 発展（慣れてきたら）

- **Skills化**：この手順書§2の内容を `.claude/skills/catelabo-skin/SKILL.md` として置くと、「新しいスキンを作って」と言うだけで手順が自動適用され、開始プロンプトのコピペが不要になる
- **ヒアリング→ブリーフ変換**：顧客のInstagram URLとヒアリングメモを渡して「ブリーフテンプレの形式に変換して」と頼めば、Step 1も半自動化できる
- **カタログの自動化**：スクショ撮影をスクリプト化（Playwright等）すれば、全スキン×全バリアントの一覧画像が1コマンドで更新できる
