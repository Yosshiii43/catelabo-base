# キャテラボ テーマ開発指針(for Claude Sonnet セッション)

このドキュメントは、キャテラボ商品テーマのCSS/テンプレート開発をAI(Claude Sonnet)と進める際に、**毎回セッション冒頭に貼り付ける規約**である。ここに書かれたルールは提案・変更の対象ではなく、遵守事項として扱うこと。

---

## 1. プロダクトの前提

- キャッテリー専門のWordPressテーマ。**1つのテーマを全顧客で共用し、スキンCSSの切り替えだけで「まったく別のサイト」に見せる**
- データ層(子猫・親猫のCPT、フィールド)は catelabo-core プラグインが持つ。テーマは表示のみ担当
- 制作者はHTML/CSS(SCSS)/JavaScript/WordPressを使用。**CSSはSCSS(FLOCSS準拠・Dart Sassでコンパイル)、JSはバニラ**で書くこと(Tailwind不可、npmはSassビルドのみに使用)
- **トークンはSCSS変数にせず、CSSカスタムプロパティで定義する**(スキンCSSを最後に読み込んで実行時に上書きする機構のため。SCSS変数はコンパイル時に消えるので使えない)
- CSS設計は **FLOCSS** に準拠する

## 2. 差別化アーキテクチャ v2(最重要)

目標は「同じデータ・同じHTML構造のまま、**誰が見ても別のサイト**」。差別化は次の4レバーの掛け算で作る。

**レバー1:デザイントークン(土台)**
色・書体・角丸・影・余白は必ず `var(--token)` 経由。コンポーネントCSS内の生HEX・font-family直書きは禁止(_tokens.scssとskins/のみ例外)。

**レバー2:セクションバリアント(配置を変える)**
主要セクションに複数レイアウトを実装し、カスタマイザー→bodyクラスで切替。
現行:ヒーロー(`p-hero--*`/catelabo_hero_variants)・子猫一覧(`klist-*`/catelabo_kitten_list_variants)・見出し装飾(`hstyle-*`/catelabo_heading_styles)。
ヘッダー・ごあいさつ・詳細・フッターにも同方式で拡張していく。**バリアント追加=SCSSに1ブロック+登録関数に1項目**が原則。レイアウト骨格の変更は必ずこのレバーで行う(スキンでやらない)。

**レバー3:モーション(動きの性格)**
bodyの `motion-{none|soft|rich}`(catelabo_motion)で出現・ホバー演出の強度を切替。出現は `c-reveal` を要素に付けるだけ(main.jsが監視)。独自のスクロール監視JSを増やさず、この仕組みに乗せる。prefers-reduced-motion時に全て無効化される設計を壊さない。

**レバー4:スキン(見た目と「構成」の自由記述)**
`assets/scss/skins/skin-{name}.scss`。行数制限なし。トークン再定義を起点に、装飾も**CSSによるレイアウトの再構成**も自由に書いてよい(gridの組み替え・writing-mode縦書き・疑似要素タイポ・出現モーションの署名変更など。見本=skin-yuki)。境界は3つ:
- **HTML・契約クラス(§5)・テンプレートPHP・JSには触れない**(構造の再構成はCSSだけで行う)
- **スマホ表示・:focus-visible・prefers-reduced-motion を壊さない**
- **トークン優先**:トークンで表現できるもの(色・書体・角丸・影)は必ずトークン再定義で
複数案件で使い回したい構成は、バリアント(レバー2)への昇格を検討する(任意)。スキンCSSは選択時のみ読み込まれるため、他スキンへの影響はない。

**デザインプリセット**:1案件の見せ方=スキン×ヒーロー×子猫一覧×hstyle×モーションの組み合わせ。ブリーフに組み合わせを記録し、カタログのスクショ名にも反映する(例: `yuki-minimal-rows-bottom-soft.png`)。

## 3. FLOCSS構成とファイル配置

```
assets/scss/
├ main.scss          エントリーポイント(@useでFLOCSS順に読み込む)
├ foundation/
│   _functions.scss  rem() / rclamp() / pc-vw() tab-vw() sp-vw()（CSS出力なし）
│   _mixins.scss     mq() / cq() / bleed-side() bleed-both() / arrow()（CSS出力なし）
│   _reset.scss      リセット
│   _tokens.scss     デザイントークンのデフォルト定義(:root・CSSカスタムプロパティ)
│   _base.scss       html/body・タイポベース
├ layout/            _l-*(コンテナ・セクション枠。**幅・余白のプリミティブのみ、装飾禁止**)
├ object/
│   component/       _c-*(最小部品:ボタン・バッジ・見出し・フォーム部品)
│   project/         _p-*(具体機能:ヘッダー・フッター・子猫カード・子猫一覧・詳細・親猫カード・ヒーロー・更新フォーム)
│   utility/         _u-*(単一目的。乱用禁止)
└ skins/
    skin-default.scss
    skin-{client}.scss   ※パーシャルにしない(単体でコンパイルされ実行時に切替)
```

関数・mixinを使うパーシャルは冒頭に次の2行を書く(object配下は `../../foundation/…`)。
`@use "../foundation/functions" as *;` `@use "../foundation/mixins" as *;`
**@media直書きは禁止、必ず `mq(sp / tab / pc)` を通す**(sp=767px以下のSP専用/tab=768px〜/pc=1024px〜)。

ビルド:テーマ直下で `sass assets/scss:assets/css --no-source-map`(Dart Sass)。
出力は `assets/css/main.css` と `assets/css/skins/skin-*.css`。functions.phpがmain→スキンの順で読み込む。
新しいコンポーネントを作ったら main.scss に `@use` を追記すること。ネスト(`&__` `&--`)は使用可、ただし3階層まで。

## 4. 命名規約

- プレフィックス:`l-` `c-` `p-` `u-`。BEMで `__`(Element)`--`(Modifier)
- 状態クラスは `is-*`(例:`is-open`)。JSフックは `js-*` とし、**js-クラスにスタイルを当てない**
- 例:`p-kitten-card p-kitten-card--decided` `c-btn c-btn--primary`

## 5. クラス契約(変更禁止リスト)

以下のクラス名とdata属性はPHPテンプレートとプラグインが出力する**固定API**。Sonnetはこれらの名称変更・削除を提案しないこと(スタイルの中身は自由)。

| 契約クラス | 意味 |
|---|---|
| `p-kitten-card` `__photo` `__status` `__name` `__meta` `__age` | 子猫カード一式 |
| `p-kitten-card[data-status="boshu\|shodan\|kettei"]` | ステータス(CSSで色分けに使用) |
| `p-kitten-list` | 一覧グリッド |
| `p-kitten-detail` `__gallery` `__spec` `__health` `__intro` `__parents` `__cta` | 詳細ページ各部 |
| `p-parent-card` | 親猫カード |
| `p-hero` +バリアントModifier | ヒーロー |
| `p-greeting` | TOPのごあいさつセクション |
| `p-header` `p-footer` | ヘッダー・フッター（ページの家具。装飾を含むためProject層） |
| `p-manage` `__list` `__form` | かんたん更新フォーム(Phase 3) |
| `l-container` `l-section` `l-main` | レイアウト枠（幅・余白のプリミティブのみ。装飾禁止） |
| `c-btn` `c-badge` `c-heading` | 基礎部品 |
| `c-reveal`(+`is-inview`) | スクロール出現。main.jsが監視(スタイルとJSフックを兼ねる例外) |

bodyクラス契約(名称変更禁止): `skin-*` `hstyle-*` `motion-{none|soft|rich}` `klist-*` (+今後のセクションバリアント)。

## 6. デザイントークン一覧(_tokens.cssの契約)

```css
:root{
  /* 色 */
  --color-bg:            /* ページ背景 */
  --color-surface:       /* カード・面 */
  --color-ink:           /* 本文 */
  --color-ink-sub:       /* 補助文字 */
  --color-primary:       /* 主アクセント(CTA等) */
  --color-primary-ink:   /* primary上の文字色 */
  --color-accent:        /* 第2アクセント(控えめに) */
  --color-line:          /* 罫線 */
  --color-status-open:   /* 募集中 */
  --color-status-talk:   /* 商談中 */
  --color-status-done:   /* ご家族決定(グレー系) */
  /* 書体(Google Fonts読込はfunctions.phpでスキン連動) */
  --font-heading:
  --font-body:
  --font-note:           /* キャプション・英字ラベル */
  /* 形・質感 */
  --radius-xs: --radius-s: --radius-m: --radius-l:
  --radius-btn:          /* ボタン専用。999pxでピル、var(--radius-xs)でかっちり */
  --shadow-card:
  --border-card:         /* 例 1px solid var(--color-line)。枠線派/影派の切替に使う */
  /* 寸法 */
  --container:           /* 最大幅 */
  --section-pad:         /* セクション上下余白 */
  --space:               /* 基準余白(8px推奨)。余白はcalc(var(--space)*n)で組む */
}
```

トークンの追加は可(この一覧の削除・改名は不可)。スキン間で雰囲気を大きく変える鍵は、色よりも `--font-heading`(明朝/丸ゴ/ゴシック)と `--radius-*`(角丸の量)と `--border-card`/`--shadow-card`(線の世界か影の世界か)の組み合わせにある。

## 7. 品質基準(全スキン共通)

- 本文16px以上・行間1.8以上(主要読者は40〜60代)
- モバイルファースト。ブレークポイントは `mq()` 経由のみ(sp: 〜767 / tab: 768〜 / pc: 1024〜)
- 見出し系コンポーネント(タイトル・見出しのクラス)は `font-family: var(--font-heading)` を**自前で宣言**する。h1〜h4の要素セレクタに依存しない(タグは意味論の都合で変わるため)
- 文字と背景のコントラスト比4.5:1以上(スキン作成時に必ず確認)
- `:focus-visible` スタイル必須/`prefers-reduced-motion` 対応必須
- 画像は width/height 属性必須(CLS対策)。子猫写真は4:3で `object-fit:cover`
- 装飾は疑似要素を優先し、HTMLに装飾目的の要素を足さない(HTML固定の原則)

```
【スキンブリーフの型】
- 顧客名/猫種/雰囲気キーワード(3語):例「上品・静けさ・雪」
- 参考:既存Instagramの世界観URL
- トークン初期値の希望(あれば):基調色/見出し書体の系統/角丸の量
- 使用レイアウトバリアント:hero=◯◯、一覧=◯◯

- 「assets/scss/skins/skin-{name}.scss を作成。:rootのトークン再定義を主体に、上書きは100行以内」
- 「クラス契約(§5)の変更禁止。生HEX・font-family直書き禁止(トークン経由)。トークンをSCSS変数に置き換えない」
```

## 9. スキン制作の技術ノート(実案件で踏んだ落とし穴)

skin-yuki制作で実際に発生したバグ。スキン制作時は毎回このリストを踏まえること。

1. **ベースの同セレクタを必ず確認する**。ベーステーマの `min-height` 等が残っていると、スキンの `height: auto` を書いてもそちらが優先され意図を壊す(例: `.p-hero__media` の `min-height: calc(100svh - 4.5rem)` が生き残り写真枠が縦長化)。上書き対象セレクタはベース側の全プロパティをgrepしてから、必要なものを明示的に打ち消す(`min-height: 0` 等)
2. **Gridのstretchに注意**。グリッドセル内の要素は行の高さまで引き伸ばされる。画像コンテナには `align-self: start` を明示する
3. **rem()にはpxを渡す**。rem換算済みの数値を渡すと1/16に二重変換される(22pxなら `rem(22)`。`rem(1.375)` は事故)
4. **radial-gradientは `circle` を明示**。省略するとタイルの縦横比に応じた楕円になる(縦長タイルで雪が縦長になった)
5. **「重ねる」配置はabsoluteよりフローで**。絶対配置の基準(containing block)の取り違えは頻出事故。グリッド+`display: contents` で横並びセルに置く方が原理的に壊れない
6. **縦書き見出しの定石**: `white-space: nowrap`(折返しは手動改行のみ)+ `font-size: min(rclamp(...), ○svh)`(高さ連動)+ 運用ルール「1行全角10文字前後」
7. **高さの配分は `align-content` と `1fr` スペーサ行で制御**。`center` は画面が高いとき上に余白が浮く。上端固定なら `start` +伸ばしたい位置に `1fr` の行を差し込む
8. **確認は必ずスーパーリロード**。ローカルでビルドしてもファイル名・バージョンが同じだとブラウザが古いCSSを使い続ける
9. **z構成はコメントで残す**(例: `inner::before` のゴースト=z1、本文=z2、写真=z-auto)。後から触る人(未来の自分・Sonnet)のため
10. **iOSのUAスタイルはフォーム部品のサイズを`!important`相当で強制する**(iOS 26のdate入力で確認)。`width: 100%` も `min-width: 0` も `!important` 付きですら勝てない(カスケード序列はUAの!important>作者の!important)。同じ宣言ブロックのpaddingは効くのにwidthだけ効かない、という症状が出たらこれを疑う。対処はUAの描画パス自体から外す `-webkit-appearance: none`(＋値が空のとき欄が潰れる副作用対策に `::-webkit-date-and-time-value { min-height: 1.9em }`)
11. **grid/flexアイテムの `min-width: auto`(デフォルト)は中身の固有最小幅より縮むことを拒否する**。iOSのフォーム部品は大きな固有最小幅を持つため、フォーム系のグリッドアイテムと入力欄には `min-width: 0` を明示する。入力欄のセレクタは型の列挙ではなく除外方式(`input:not([type="radio"]):not(...)`)にして、プラグイン側のフィールド型追加で漏れない構造にする
12. **CSSを差し替えたらテーマバージョン(`CATELABO_BASE_VER`＋style.cssのVersion)を上げる**。`?ver=` が同じだと実機は古いCSSを配信し続け、「直したのに変わらない」でデバッグが汚染される(§9-8のスーパーリロードは自分の環境しか救わない。実機・顧客環境はバージョンで殺す)
13. **横方向のはみ出し診断は `scrollWidth` では不十分**。`overflow-x: clip` 下でも要素の実ジオメトリははみ出したままで、iOSはそれを検知して横パン(横揺れ)とズームアウトを許す。診断は全要素の `getBoundingClientRect()` 走査＋`visualViewport.scale` 監視(iOSは「はみ出し全体が収まる倍率」までしかズームアウトを許さないため、scale値から犯人の実幅を逆算できる: 例 390÷0.975=400px)
14. **CSSチャンネル名（relative color syntaxの `l` 等）を `calc()`/`min()` に入れるときは `#{'...'}` で素通しにする**。Sass処理系によってはSassScriptの計算式として解析してビルドエラーになる。文字列補間なら中身を解析せずそのままCSSに出力される（出力は同一）
15. **「保存すると」Sassエラーが出るのにCLIビルドは通る場合、DartJS Sass Compiler拡張の再有効化を疑う**。VS Codeの更新でワークスペースの無効化設定が戻ることがあり、保存時に内蔵の別バージョンdart-sassが走ってCLIと違う結果を出す。対処：拡張を「無効にする（ワークスペース）」→VS Code完全再起動（§ビルド環境の項も参照）
16. **配布ファイルの上書き後にwatchが再ビルドしないことがある**。ダウンロードしたファイルは過去のタイムスタンプを持ち、Finderの「置き換え」はそれを保持する。`sass --watch` は初回スキャンで「出力CSSの方が新しい」と判断してスキップするため、修正が反映されない。対処：上書き後に一発ビルド（`sass assets/scss:assets/css --no-source-map`、無条件フルコンパイル）を打つか、VS Codeで該当ファイルを開いて `Cmd+S`（タイムスタンプ更新＋watch発火）
17. **フォーカスリングの外描きは overflow: hidden に切られる**
    ベースの `:focus-visible` は `outline-offset: 3px` で要素の外側にリングを描く。
    角丸クリップ等で `overflow: hidden` を持つカードの内側にフォーカス対象
    （summary・ボタン等）を置くと、リングがクリップされて「不可視」や
    「一辺だけ切れ端が見える」事故になる（例: c-faq）。
    カード内のフォーカス対象は `outline-offset` を負値にしてリングを内側に描き、
    `border-radius` を親のクリップに沿わせる。
    また、リング色（primary）と同色の面（CTA帯等）の上ではリングが同化するため、
    その面の「上に載る文字色」でリング色を上書きする（例: _p-cta.scss）。
